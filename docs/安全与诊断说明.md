# 安全与诊断说明

Gmail IMAP 应用专用密码、QQ/163 授权码和 Generic IMAP/SMTP 凭据保存在当前用户 Windows Credential Manager。兼容账号保留旧 key，新账号使用不含邮箱明文的 `account:<account_id>:<secret-kind>` key；同 Provider 多账号不会共享秘密，IMAP 与 SMTP 使用独立 secret kind。界面不回显旧值，新值保存失败时保留旧凭据。

Gmail OAuth 位于 `%LOCALAPPDATA%\AgentMailBridge\OAuth`，新账号使用 `accounts/<account_id>/` 专属 credentials 与 Token。GUI 只导入用户明确选择且严格验证为 Google Desktop installed app 的 `credentials.json`；替换失败保留旧配置。清除、重新授权或失败只影响当前账号。Gmail scope 必须且只能是 `https://www.googleapis.com/auth/gmail.readonly`；因此 Gmail send 在 v1.4.1 保持未实现。

OAuth 授权和 Gmail API 网络请求不在 Qt GUI 主线程执行。本地回调只监听 `127.0.0.1` 随机端口，使用单调时钟超时、可唤醒取消、唯一 state、同进程会话防重和跨进程文件锁；成功、失败、取消、超时和应用退出均关闭服务器并释放端口。当前 OAuth 子流程会在不覆盖原代理配置的前提下补充 `127.0.0.1`、`localhost` 和 `::1` 代理绕过项。

Token 只在回调 state 校验成功后交换。Token 与当前 Client ID、`gmail.readonly` 和 refresh token 绑定检查通过后，使用同目录临时文件、flush、fsync 和原子替换保存；新授权失败、账号不匹配或写入失败不会提前删除旧 Token。清除本地 Token 必须由用户确认，且不删除 credentials。Token 交换成功与 Gmail Profile 验证成功是两个状态；API 未启用或临时网络错误可保留 Token 后单独重新验证。

OAuth 日志只允许记录阶段、Desktop 类型验证结果、127.0.0.1 回环是否绑定、是否收到回调、端口是否释放、浏览器打开结果、错误码和耗时。授权 URL、query、state、code、Client Secret、access token、refresh token、完整回调 URL和含密码的代理 URL不得进入日志、SQLite、诊断导出或报告。

自动化测试必须同时隔离配置读取与配置写入。仅禁用 dotenv 加载不能阻止默认保存路径写入源码 `.env`；全局测试夹具必须把应用运行目录和配置文件显式重定向到每项测试的临时目录。涉及默认保存路径的测试应有独立回归，并在完整测试前后校验真实配置文件未变化，但不得输出配置内容或摘要值。

安装目录只放程序与只读资源。配置、OAuth、数据、SQLite、日志、邮件、附件和备份位于当前用户可写目录。普通卸载不删除用户数据、OAuth 或 Credential Manager 凭据。

`mail_accounts` 只保存 provider、规范化地址、认证类型、能力、生命周期和非秘密 Provider 设置，不保存密码、授权码、OAuth URL、Client Secret 或 Token。软移除默认保留全部 ownership；凭据和 Token 清理由用户明确选择。旧库升级前先在线备份，迁移不移动邮件文件、不修改 raw 或历史 Hash，失败整体回滚。

每封新邮件在独立 staging 目录完成 raw、正文、资源和 manifest 后再原子提升为正式 package。`raw.eml` 只保存 Gmail API raw 或 IMAP `BODY.PEEK[]` 实际字节并记录 SHA-256；历史邮件没有原始字节时明确记为不可恢复，不生成近似 raw。manifest 只使用 package 相对路径，所有资源路径都必须位于该邮件根目录。

链接识别完全离线。可信域列表默认为空，因此普通网页、外部图片、云文档和文件样式链接都不会被访问。可信下载仅接受 HTTPS 直接文件，并对主机解析出的全部地址、连接目标和每次重定向重新执行公网 IP 检查，拒绝 localhost、私网、回环、链路本地、组播、保留地址和云元数据目标。下载使用固定解析地址与证书主机名校验，同时限制超时、重定向次数、Content-Length、流式实际大小和 MIME；只原子保存到当前 package 的 `downloads`，不会执行或解压。

诊断入口：

- Gmail API、Gmail IMAP、QQ SMTP 单项测试位于对应账号页。
- 收件页右侧“连接健康”以五个独立状态项聚合 Gmail、QQ SMTP、MCP、凭据/OAuth 和 SQLite/数据目录，记录最近检查时间并提供“去处理”。
- 收件页“管理日志”支持概览、级别/事件类型/时间/关键词组合筛选、日常检查开关、分页加载、当前筛选脱敏导出、保留设置、安全清理、完整脱敏诊断导出和日志目录。
- 设置 > 高级设置保留网络、路径、迁移、全部连接诊断、脱敏报告和最近错误。
- 文件与数据 > 数据维护负责备份、验证、恢复、一致性扫描和维护报告；恢复必须确认并先备份当前数据库。

源码 CLI 继续提供 Gmail、QQ、网络、凭据和数据诊断。构建前必须执行秘密扫描；`.env`、credentials.json、token.json、数据库、日志、邮件和附件不得进入 Git、dist、release 或报告。

收件结果的安全语义固定为 success、no_changes、partial、failed。正常无新邮件使用 no_changes，不写 ERROR、不增加失败统计；单项处理失败但已保存部分内容使用 partial 和 WARNING；只有认证、网络、连接或主流程不可执行等真实故障才使用 failed 和 ERROR。

技术日志有两套独立边界。文件日志 `app.log` 按约 2 MB 单文件、5 个备份轮转；SQLite `app_events` 默认普通 INFO/SUCCESS 保留 30 天、WARNING/ERROR/FAILED 保留 90 天，最多 10,000 条，超限后按时间顺序批量降到约 8,000 条。应用启动后异步执行一次清理，持续运行每 24 小时执行一次，事件插入检测到明显超限时执行补充数量治理，用户也可手动清理。自动治理只做轻量事务，不自动 VACUUM；VACUUM 只属于用户主动数据库维护边界。

周期自动检查开始、正常健康轮询、重复扫描和无新邮件不再永久写入 `app_events`，这些状态由账号级 `account_sync_states`、v1.3 兼容 `auto_receive_state` 和收件状态卡保存。收到并保存新邮件、部分内容失败、needs_attention、认证/网络/数据库/文件异常、进入或退出全局退避、连接恢复、发件、MCP success/duplicate/failure、授权/配置变化、备份恢复、一致性扫描和日志清理仍会记录。用户主动“立即收取”仍可记录开始和结果。

日志时间清理和数量清理只执行 `DELETE FROM app_events`，不会级联或触碰 `mail_packages`、`mail_resources`、`received_messages`、`received_files`、`outbound_messages`、`outbound_resources`、`sent_files`、`mcp_calls`、`receive_retries`、raw.eml、manifest 或附件。清理失败只写文件诊断，不递归制造新的 SQLite 日志风暴。当前筛选导出会掩码邮箱、绝对 Windows 路径和 token/password/auth/secret 形式的值，不导出邮件正文或附件内容。

MCP stdout 只承载 UTF-8 JSON-RPC，stderr 和文件日志负责诊断。提交文件必须先通过 `DATA_ROOT`/用户授权工作区白名单，再由程序原子 staging；白名单不会因为 GUI 曾选择全局文件而扩大。GUI 用户可为手动邮件填写一个合法收件人，但 MCP `submit_result` schema 不接受 recipient，始终受 `OWNER_GMAIL` 控制。授权入口拒绝磁盘根目录、用户主目录、Windows、Program Files、ProgramData、AppData 和产品数据目录，提交入口额外拒绝 `.env`、credentials.json、token.json。`mcp_calls` 审计 staging 状态、源/暂存大小和 SHA-256、失败原因；`sent_files` 保存 pre-SMTP 与归档 Hash。任何发送前 Hash 不一致都会阻止发信。

v1.2.0 的邮件读取是全局一次性 opt-in，默认关闭，不使用逐封分享令牌。读取工具只允许访问数据库登记且再次验证位于 `DATA_ROOT` 内的正式 package；资源必须属于请求的邮件，路径、大小和已有 SHA-256 会在访问时重新核验。即使数据库路径被篡改，也不能越过 DATA_ROOT。正文与资源读取有字符或行数上限和分页参数；raw 模式只允许读取实际归档的 `raw.eml`。

`prepare_mail_resources` 只写入 `ALLOWED_SEND_ROOTS` 中明确授权且仍可用的工作区。目标目录逐级解析，预置符号链接或 Windows 目录联接不能把写入导向工作区外；复制采用同目录临时文件和原子替换，并在完成后核对大小与 SHA-256。图片、PDF、Office、压缩包和可执行文件不会在桥接器中执行或解压。

`mcp_audit_events` 统一记录 search/get/read/prepare/workspace/sync/send 的工具名、时间、客户端、目标摘要、状态、错误码、计数、耗时和必要 Hash 事实；查询文本和路径按上限保存，邮件正文全文、附件内容、凭据和 OAuth 不进入审计。旧 `mcp_calls` 保留，查询层合并显示并避免重复发送记录。

自动收件把账号级连接失败与单邮件失败分开。连接失败只更新该账号调度退避；单邮件失败带 `account_id` 进入 `receive_retries`，依次等待 1 分钟、5 分钟、30 分钟、2 小时，继续失败后转为 needs_attention。未到期或终态项目不会重复处理，避免一个账号的坏邮件污染其他账号。`account_sync_states` 持久化各账号的启用状态、周期、检查/成功/结果/错误、连续失败、下次检查和不含秘密的 mailbox UID checkpoint。Generic IMAP 只执行 `BODY.PEEK[]` 和可选用户明确触发的 Seen 标记；默认收取和历史补扫不改变服务器已读状态。

历史补扫只由 GUI 用户明确启动，日期范围、page size 和 scan cap 有上限，逐封处理且可取消；与自动收件共用进程内和跨进程锁。它不删除邮件、不执行附件、不放宽可信下载，Gmail API 仍是 `gmail.readonly`，IMAP 使用 `BODY.PEEK[]`。规则评估表只保存 provider/message id、结果、原因、非敏感规则指纹和 scan id，不保存正文或附件，也不参与永久去重。

本机发送使用 `X-AgentMailBridge-Origin` 和随机 `X-AgentMailBridge-Outbound-ID` 标记，不含凭据。回流识别还必须命中本地 outbound 记录和实际发件身份；仅仅 From 与 QQ 地址相同不会被认作本机发送。防重复发送继续由 request_id/outbound 层负责，收件归档规则不再承担防回流。

收件规则只保存非秘密配置。v1.3.0 新配置默认 `all_scanned`；没有显式模式、仅继承旧 `AUTO_RECEIVE_ONLY_SELF_MAIL=true` 默认的配置会带 schema version/source 标记一次性迁移到 `all_scanned`。显式 `self_only` 和 custom 规则保留；`self_only` 的准确含义是“仅 Gmail 自发自收邮件（高级）”。迁移原子、幂等，失败保留旧配置。所有收件入口共用同一规则，不改变 OAuth scope、凭据存储或扫描后端权限。

坏邮件重试状态保存在 SQLite；连接级故障不进入该表。到期项可脱离普通回看窗口按 Gmail 资源 ID 或 IMAP UID 重试，第五次仍失败后进入 `needs_attention`，后续健康轮询不再处理该项。

文件与数据把 `mail_resources` 作为新收件资源权威来源，`received_messages` 只承担旧业务兼容，未映射的 `received_files` 继续显示。旧发送记录的大小补全仅允许对 `DATA_ROOT` 或授权工作区内的路径执行 `stat`，不读取文件内容、不越过信任根目录，单个文件缺失也不会中断页面。发件内容由 `outbound_messages`、`outbound_resources`、`outbound_links` 统一表达，一次发送对应一个 outbound；旧 `sent_files` 会幂等回填但不伪造正文。

一致性扫描同时核对邮件 package、manifest、raw、正文和本地资源是否存在，检查路径是否越界、大小和 SHA-256 是否一致，并报告残留 staging 或孤立目录；扫描只报告和更新可观察状态，不静默删除邮件或附件。

当前 EXE 与安装器未签名时，Authenticode 状态为 NotSigned，SmartScreen 可能提示未知发布者。不得绕过提示或伪造签名。
