# 统一邮件归档设计

## 目标与边界

本阶段把“一封邮件”建立为可长期核验的本地事实对象。它不是通用邮件客户端、知识管理系统、多租户服务或多邮箱 Route B，也不改变 v1.0.0 现有 GUI 信息架构。

一封邮件只允许存在一个正式 package。正文、内嵌图片、普通附件、链接和下载资源必须关联同一个 `package_id`；兼容表只服务现有界面，不再是完整邮件事实的权威来源。

## 身份模型

- `package_id`：由规范化 `account_ref` 与规范化 Message-ID 计算稳定 SHA-256 摘要，重复扫描和 Gmail API/IMAP 后端切换不会产生第二个包。
- `account_ref`：当前 Gmail 账号的稳定引用；未知历史数据使用明确 legacy fallback，不硬编码“永远只有一个账号”。
- `mailbox_ref`：记录 Gmail/IMAP 的邮箱范围，为未来路由保留边界。
- `thread_ref`：Gmail 优先使用 provider thread id；IMAP 只使用明确的 References/In-Reply-To，无法确认时安全退化，不按主题猜测会话。
- `resource_id`：由 package、资源类型、来源和稳定序号生成，同一邮件重试时保持可关联。

## 目录与生命周期

正式目录为 `received/mail/YYYY/MM/DD/<package_id>_<subject-slug>`。主题和文件名经过非法字符清理、冲突消解与路径预算控制，数据库和 manifest 仍保存完整显示名与原始名。

新邮件先在 `received/mail/.staging/<package_id>.tmp` 完成全部写入。文件使用临时文件、flush、fsync 和原子替换；新包完成后再原子提升到正式目录。部分失败会保留同一 package 的成功资源和错误状态，重试只补写失败或缺失项，不另建目录。

## 原始邮件与正文

- Gmail API 使用 `messages.get(format=raw)` 返回的 base64url 内容，解码后原样写入 `raw.eml`。
- IMAP 使用 `BODY.PEEK[]` 返回的 RFC822 原始字节，原样写入 `raw.eml`。
- `raw.eml` 的 SHA-256、大小和状态进入数据库与 manifest。新邮件没有真实 raw 时不能成为 ready 包。
- 历史数据没有原始字节时只写 `legacy_missing` 状态和说明，不创建近似 `raw.eml`。
- 正文分别保存 `body.txt`、`body.html` 和安全可读的 `body.md`。HTML 转换离线执行，忽略脚本、样式、head、noscript 与 template 内容。

## 资源模型

`mail_resources` 记录资源类型、来源、显示名、原始名、MIME、Content-ID、URL、相对路径、大小、SHA-256、状态和错误。

- 带 CID 且在 HTML 中引用的图片归类为 `inline_image`。
- disposition 为 attachment 或带普通文件名的图片归类为 `attachment`。
- 普通附件保留在 `attachments`，同名文件使用稳定冲突名。
- plain URL、HTML href 和非 CID img src 离线识别为网页、文件样式链接、云文档或外部图片链接。
- 普通链接和云文档只记录，不访问；下载成功的资源仍属于原 package 并保存到 `downloads`。

## 可信下载

可信域列表位于 SQLite，默认为空。规则区分精确域和显式允许子域，不使用字符串后缀模糊匹配。

下载要求 HTTPS。解析阶段拒绝非公网、localhost、私网、回环、链路本地、组播、保留、未指定和元数据目标；所有 DNS 结果都必须安全。连接固定到已校验 IP，同时保留 TLS SNI 和证书主机名校验；每次重定向重新校验协议、域名、DNS 和地址。Content-Length 与实际流式字节分别受限，超时、重定向次数、MIME、Content-Disposition 文件名和最终 SHA-256均受审计。下载文件不会执行或解压。

## Manifest

每个包的 `manifest.json` 包含 `schema_version`、身份、账号/邮箱/会话、元数据、raw、正文、资源清单、分类计数、解析/归档状态、错误和时间。所有本地文件路径均为 package 相对路径；URL 是资源事实，不作为本地路径处理。

## 数据库与迁移

`mail_packages` 与 `mail_resources` 是邮件和资源权威事实表；`received_messages` 与 `received_files` 保持现有 GUI、历史和文件管理兼容，并通过 `package_id/resource_id` 回指权威对象。

启动时若检测到旧库，先创建并校验 SQLite 在线备份，再执行 schema 增量迁移和 legacy backfill。历史正文/附件复制到独立 legacy package，原文件不移动、不删除；迁移元数据保证重跑幂等。迁移可以区分旧数据真实 0 字节、未知大小和缺失文件。

## 一致性与恢复

一致性扫描检查数据库 package 与目录对应关系、manifest 结构和 package_id、raw 与资源存在性、相对路径越界、大小/SHA-256、孤立 package 和残留 staging。扫描不静默删除用户数据。数据库备份/恢复沿用在线备份、Hash 清单、恢复前安全备份和失败回滚策略。

## 兼容策略

自动收件、Gmail API/IMAP 规则、有限重试和 `no_changes/partial` 语义不变。v1.2.0 在规范归档之上增加 Agent 只读访问：每次访问重新验证 package 位于 `DATA_ROOT`、资源属于该 package 且路径不逃逸；正文和资源读取有界分页。资源准备只把归档副本写入授权工作区，不修改规范归档。当前收件、历史和“文件与数据”继续读取兼容 DTO。
