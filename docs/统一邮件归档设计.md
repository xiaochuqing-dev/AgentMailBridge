# 统一邮件归档设计

## 目标与边界

本设计在 v1.4.2 Multi-Account Runtime 中继续把“一封邮件”建立为可长期核验的本地事实对象，并让 `account_id` 同时成为运行路由和正式归属。Gmail 与共享 Generic IMAP Core 都只能把真实取得的原始邮件交给同一归档流程；QQ、163 或 Generic 不另建归档格式。它不是通用邮件客户端、知识管理系统或多租户服务。

一封邮件只允许存在一个正式 package。正文、内嵌图片、普通附件、链接和下载资源必须关联同一个 `package_id`；兼容表只服务现有界面，不再是完整邮件事实的权威来源。

## 身份模型

- `package_id`：由规范化 `account_ref` 与规范化 Message-ID 计算稳定 SHA-256 摘要，重复扫描和 Gmail API/IMAP 后端切换不会产生第二个包。
- `provider_message_id`：保存 Gmail message id 或 IMAP UID；数据库以账号、后端和 provider id 建立第二层唯一事实。异常邮件的 RFC Message-ID 改变时，历史补扫也不会产生第二个包。
- `account_id`：由 provider 与规范化地址生成的稳定不透明身份，是正式账号外键，也是凭据、OAuth、同步、重试、进程锁和 Provider Adapter 的运行边界；`account_ref` 仅作为 v1.3 兼容字段保留。
- `mailbox_id`：由 `account_id` 与 Provider 外部邮箱目录引用生成；`mailbox_ref` 保留 Gmail/IMAP 原始范围。
- `thread_ref`：Gmail 优先使用 provider thread id；IMAP 只使用明确的 References/In-Reply-To，无法确认时安全退化，不按主题猜测会话。
- `resource_id`：由 package、资源类型、来源和稳定序号生成，同一邮件重试时保持可关联。

## 目录与生命周期

正式目录为 `received/mail/YYYY/MM/DD/<package_id>_<subject-slug>`。主题和文件名经过非法字符清理、冲突消解与路径预算控制，数据库和 manifest 仍保存完整显示名与原始名。

新邮件先在 `received/mail/.staging/<package_id>.tmp` 完成全部写入。文件使用临时文件、flush、fsync 和原子替换；新包完成后再原子提升到正式目录。部分失败会保留同一 package 的成功资源和错误状态，重试只补写失败或缺失项，不另建目录。

账号停用或软移除不会移动或删除 package，也不会改写 manifest、raw.eml 或 Hash。恢复同一 provider + 地址会得到相同稳定账号身份；是否删除该账号 Credential/OAuth 由用户单独选择，归档 ownership 始终保留。

## 原始邮件与正文

- Gmail API 使用 `messages.get(format=raw)` 返回的 base64url 内容，解码后原样写入 `raw.eml`。
- IMAP 使用 `BODY.PEEK[]` 返回的 RFC822 原始字节，原样写入 `raw.eml`。
- `raw.eml` 的 SHA-256、大小和状态进入数据库与 manifest。新邮件没有真实 raw 时不能成为 ready 包。
- 历史数据没有原始字节时只写 `legacy_missing` 状态和说明，不创建近似 `raw.eml`。
- 正文分别保存 `body.txt`、`body.html` 和安全可读的 `body.md`。HTML 转换离线执行，忽略脚本、样式、head、noscript 与 template 内容。
- From、To、Cc、Bcc、Reply-To 的原始 Header 与 RFC 2047 解码后的显示名、地址同时保存；`raw.eml` 永不为展示目的重写。一般邮箱地址只做解析，Gmail canonicalization 仅用于 Gmail 身份判断。

## 资源模型

`mail_resources` 记录资源类型、来源、显示名、原始名、MIME、Content-ID、URL、相对路径、大小、SHA-256、状态和错误。

- 带 CID 且在 HTML 中引用的图片归类为 `inline_image`。
- disposition 为 attachment 或带普通文件名的图片归类为 `attachment`。
- 普通附件保留在 `attachments`，同名文件使用稳定冲突名。
- plain URL、HTML href 和非 CID img src 离线识别为网页、文件样式链接、云文档或外部图片链接。
- 链接显示名综合有意义的 anchor、链接类型、已知 provider、hostname 和路径文件名；识别不依赖正文出现“链接”“URL”或“下载”等提示词，`cid:` 永不当作外链。
- 普通链接和云文档只记录，不访问；下载成功的资源仍属于原 package 并保存到 `downloads`。

## 可信下载

可信域列表位于 SQLite，默认为空。规则区分精确域和显式允许子域，不使用字符串后缀模糊匹配。

下载要求 HTTPS。解析阶段拒绝非公网、localhost、私网、回环、链路本地、组播、保留、未指定和元数据目标；所有 DNS 结果都必须安全。连接固定到已校验 IP，同时保留 TLS SNI 和证书主机名校验；每次重定向重新校验协议、域名、DNS 和地址。Content-Length 与实际流式字节分别受限，超时、重定向次数、MIME、Content-Disposition 文件名和最终 SHA-256均受审计。下载文件不会执行或解压。

## Manifest

每个包的 `manifest.json` v2 包含 `schema_version`、`account_id`、`mailbox_id`、兼容引用、会话、元数据、raw、正文、资源清单、分类计数、解析/归档状态、错误和时间。所有本地文件路径均为 package 相对路径；URL 是资源事实，不作为本地路径处理。

## 数据库与迁移

`mail_accounts`、`mailboxes` 和 `account_sync_states` 保存账号、邮箱目录与每账号同步事实；`mail_packages` 与 `mail_resources` 是邮件和资源权威事实表。`received_messages` 与 `received_files` 保持现有 GUI、历史和文件管理兼容，并通过 `account_id/package_id/resource_id` 回指权威对象。同一 Message-ID 可在不同账号中分别存在，但同一账号内仍去重。

启动时若检测到旧库，先创建并校验 SQLite 在线备份，再在同一可回滚事务中执行建表、schema 增量迁移、Gmail/QQ 账号映射、ownership 回填和唯一索引创建。该迁移只写数据库，不移动文件、不改写 raw 或 Hash，并可安全重复执行；失败时事务整体回滚。联系人、outbound origin 和轻量规则评估事实随后从真实 `raw.eml` 或旧 Header 幂等回填，不修改原文。`receive_rule_evaluations` 只用于审计最近一次匹配结果，绝不作为未来补扫的永久黑名单。

## 一致性与恢复

一致性扫描检查数据库 package 与目录对应关系、manifest 结构和 package_id、raw 与资源存在性、相对路径越界、大小/SHA-256、孤立 package 和残留 staging。扫描不静默删除用户数据。数据库备份/恢复沿用在线备份、Hash 清单、恢复前安全备份和失败回滚策略。

## 兼容策略

v1.4.2 保留 Gmail API、Gmail IMAP、共享 Generic IMAP、自动、立即收取和 MCP 增量同步的共享规则；明确账号调用由 Account Runtime Router 使用该账号的配置、凭据、OAuth、锁和同步状态。历史补扫独立于增量 lookback/UID checkpoint，按账号及 Message-ID/provider id 去重；`rule_skipped` 可在规则变化后重新评估。Agent 访问仍只读、有界且受 `DATA_ROOT`/工作区边界控制，七工具和兼容 DTO 保持。
